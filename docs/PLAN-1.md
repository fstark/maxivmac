# Phase 1 — CMake Build System (Detailed Plan)

Replace the 3-stage build pipeline with a single CMakeLists.txt that directly compiles sources against the existing `cfg/` headers. The old build scripts and `setup/` directory remain untouched.

---

## Context: How the Build Works Today

1. `gcc -o setup_t setup/tool.c` — compiles the setup tool
2. `./setup_t [flags...] > setup.sh` — generates a shell script
3. `. ./setup.sh` or `bash setup.sh` — writes 7 config headers into `cfg/` and a platform build file (Makefile or Xcode project)
4. `xcodebuild` or `make` — compiles the emulator

The generated Xcode project compiles 18 source files from `src/`, links AppKit + AudioUnit + OpenGL, and includes `src/` and `cfg/` in the header search path. The 7 config headers in `cfg/` are already checked into the repo (for the Mac II / Cocoa / macOS target). The `.gitignore` excludes `cfg/` but the files are tracked anyway.

**Key insight:** Since `cfg/` already contains valid headers for the macOS/Cocoa/Mac II variant, we can just write a CMakeLists.txt that compiles against them *as-is* — no templating needed for the first working build. Templating comes as a refinement.

## Steps

### Step 1.1 — Minimal CMakeLists.txt for macOS/Cocoa

Create `CMakeLists.txt` that replicates what the Xcode project does:

**Sources** (18 compilation units — exactly matching the Xcode project):
```
src/OSGLUCCO.m    (Objective-C — Cocoa backend)
src/GLOBGLUE.c
src/M68KITAB.c
src/MINEM68K.c
src/VIAEMDEV.c
src/VIA2EMDV.c
src/IWMEMDEV.c
src/SCCEMDEV.c
src/RTCEMDEV.c
src/ROMEMDEV.c
src/SCSIEMDV.c
src/SONYEMDV.c
src/SCRNEMDV.c
src/VIDEMDEV.c
src/MOUSEMDV.c
src/ADBEMDEV.c
src/ASCEMDEV.c
src/PROGMAIN.c
```

Note: Files not in the Xcode project but in `src/` (SNDEMDEV.c, KBRDEMDV.c, PMUEMDEV.c, DISAM68K.c) can safely be added — they compile to empty translation units when their guard defines are 0. Including them means the source list doesn't need to change per-variant.

**Include paths:**
- `src/` (source headers)
- `cfg/` (generated config headers)

**Frameworks** (macOS only):
- AppKit
- AudioUnit
- OpenGL

**Build settings to replicate:**
- `-Os` optimization
- `-Wall -Wundef -Wno-uninitialized`
- C language (not C++, yet — Phase 2 handles that)
- `MACOSX_DEPLOYMENT_TARGET = 10.15`

**App bundle:**
- `MACOSX_BUNDLE` with `cfg/Info.plist`
- Bundle resource: `src/ICONAPPO.icns`
- Bundle resource: `cfg/English.lproj/dummy.txt`

**Verification:** `cmake -B bld -G Ninja && cmake --build bld` produces `minivmac.app` in `bld/`. Boot a System 7 disk image to confirm it works.

### Step 1.2 — Add SDL backend support (Linux/Windows/macOS-SDL)

Add a CMake option to select the backend:

```cmake
option(MINIVMAC_BACKEND "Platform backend" "cocoa")  # cocoa | sdl
```

When `MINIVMAC_BACKEND=sdl`:
- Compile `src/OSGLUSDL.c` instead of `src/OSGLUCCO.m`
- Swap `cfg/CNFUIOSG.h` to define `WantOSGLUSDL 1` instead of `WantOSGLUCCO 1`, and `#include <SDL3/SDL.h>` (or SDL2) instead of Cocoa headers
- Find SDL via `find_package(SDL3)` or `find_package(SDL2)`
- Drop AppKit/AudioUnit/OpenGL frameworks, link SDL instead

**Approach:** Use `configure_file()` for `CNFUIOSG.h` only — it's the one header that differs between backends. The other 6 config headers stay as-is (they're backend-independent for same model/resolution).

Create `cfg/CNFUIOSG.h.in`:
```c
/* Auto-generated by CMake — do not edit */
@CNFUIOSG_SYSTEM_INCLUDES@
#define EnableDragDrop 1
#define MyAppIsBundle 1
#define WantOSGLU@MINIVMAC_BACKEND_UPPER@ 1
#define kStrAppName "Mini vMac"
#define kAppVariationStr "minivmac-cmake"
#define kStrCopyrightYear "2026"
#define kMaintainerName "@MINIVMAC_MAINTAINER@"
#define kStrHomePage "@MINIVMAC_HOMEPAGE@"
```

CMake sets:
- `MINIVMAC_BACKEND_UPPER` = `CCO` or `SDL`
- `CNFUIOSG_SYSTEM_INCLUDES` = the appropriate `#import`/`#include` block

**Verification:** On Linux: `cmake -B bld -DMINIVMAC_BACKEND=sdl && cmake --build bld` produces a working `minivmac` binary.

### Step 1.3 — CMake Presets

Create `CMakePresets.json` with convenient presets:

| Preset | Backend | Notes |
|--------|---------|-------|
| `macos-cocoa` | Cocoa | Default on macOS. Ninja generator. |
| `macos-sdl` | SDL3 | For testing SDL on macOS. |
| `linux-sdl` | SDL3 | Default on Linux. |
| `windows-sdl` | SDL2 | Default on Windows (SDL2 has better MinGW support). |

Usage: `cmake --preset macos-cocoa && cmake --build --preset macos-cocoa`

### Step 1.4 — Expose Key Config Knobs as CMake Options

Template the remaining config headers so the build can target different Mac models without editing files manually. The 5 remaining config headers become `.h.in` templates in a `cmake/` directory:

| Config Header | CMake Variables Exposed |
|---------------|------------------------|
| `CNFUDPIC.h` | `MINIVMAC_MODEL` (Plus/SE/II/IIx), `MINIVMAC_CPU` (68000/68020), `MINIVMAC_FPU` (on/off), `MINIVMAC_RAM` (size), `MINIVMAC_EMU_*` device enables |
| `CNFUDALL.h` | `MINIVMAC_SCREEN_WIDTH`, `MINIVMAC_SCREEN_HEIGHT`, `MINIVMAC_SCREEN_DEPTH`, `MINIVMAC_SOUND`, `MINIVMAC_NUM_DRIVES` |
| `CNFUDOSG.h` | `MINIVMAC_ROM_FILE`, `MINIVMAC_MAGNIFY`, `MINIVMAC_FULLSCREEN`, `MINIVMAC_SPEED` |
| `CNFUIALL.h` | No changes needed — compiler-detection logic is static |
| `CNFUIPIC.h` | No changes needed — empty |
| `STRCONST.h` | `MINIVMAC_LANGUAGE` (English/French/German/etc.) |

The tricky part is `CNFUDPIC.h` — the Wire enum and VIA configuration are model-dependent. Strategy:

- Create a `cmake/models/` directory with per-model wire/VIA config fragments:
  - `cmake/models/MacII.h.in` — Wire enum + VIA config for Mac II
  - `cmake/models/MacPlus.h.in` — Wire enum + VIA config for Mac Plus
  - etc.
- `CNFUDPIC.h.in` includes the appropriate model fragment via `configure_file()` or generates the full file with CMake logic.
- **Alternative (simpler):** Keep pre-written `CNFUDPIC.h` files per model in `cmake/models/` and copy the right one. The Wire/VIA config is complex enough that templating it isn't worth it — there are only 12 models.

**Recommended approach:** Pre-written `CNFUDPIC.h` per model, selected by `MINIVMAC_MODEL`. CMake copies it to the build tree and overrides specific values (RAM size, extensions) via `#undef`/`#define` in a wrapper header, or via a simple Python/CMake script. This avoids having to replicate the setup tool's model→wire mapping logic.

**Simplest first pass:** Ship with just Mac II config (as-is). Add Plus and IIx as the next two. The full set of 12 models is a later task.

### Step 1.5 — Update .gitignore

Add the CMake build directory:
```
/bld/
/build/
CMakeUserPresets.json
```

Remove `/cfg` from `.gitignore` if we want to keep the checked-in config headers as fallbacks (they're already tracked despite being in `.gitignore`). Or leave as-is — CMake generates its own copies in the build tree.

### Step 1.6 — Verify & Document

1. **macOS Cocoa build:** `cmake --preset macos-cocoa && cmake --build --preset macos-cocoa` → boot System 7
2. **Linux SDL3 build:** (on a Linux box or CI) `cmake --preset linux-sdl && cmake --build --preset linux-sdl` → boot System 7
3. Add a brief `docs/BUILDING.md` explaining the new CMake build as the primary method, with the old `build_*.sh` scripts noted as legacy.

## Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| Xcode project has implicit settings we miss | Compare `xcodebuild -showBuildSettings` output with our CMake flags |
| SDL `find_package` doesn't work everywhere | Provide `SDLDIR` hint variable; fall back to pkg-config |
| Some `.c` files rely on include order from the Xcode project | Unlikely — each `.c` file explicitly includes `PICOMMON.h` or `OSGCOMUI.h` as its first include, which chains all config headers. No implicit dependencies. |
| Config header templating for all 12 models is complex | Defer full model support. Ship with Mac II (already working). Add Plus and IIx next. Other models are rare use cases. |
| OpenGL is deprecated on macOS | It still works (and the Xcode project uses it). Address in a later phase (Metal backend). |
