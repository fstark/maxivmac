# HFS Trap Variants — Detailed Design

Make the trap tracer display HFS File Manager calls (HOpen, HGetVol,
etc.) as distinct named traps with their HFS-specific fields visible.

All code follows [STYLE.md](STYLE.md) and [NAMING.md](NAMING.md).

---

## Background

Classic Mac OS File Manager traps come in two flavours, documented
side by side in Inside Macintosh (File Manager chapter):

| Flat (MFS) | HFS variant | Trap word |
|------------|-------------|-----------|
| Open       | HOpen       | A000 / A200 |
| Create     | HCreate     | A008 / A208 |
| Delete     | HDelete     | A009 / A209 |
| OpenRF     | HOpenRF     | A00A / A20A |
| Rename     | HRename     | A00B / A20B |
| GetFileInfo| HGetFileInfo| A00C / A20C |
| SetFileInfo| HSetFileInfo| A00D / A20D |
| GetVol     | HGetVol     | A014 / A214 |
| SetVol     | HSetVol     | A015 / A215 |

From Inside Macintosh:

> HOpen is an extension of the Open call that lets you use a directory
> ID and a pathname to specify the file to be opened 
> These hierarchical routines use the record HParamBlockRec, which
> is a superset of ParamBlockRec.

And in the assembly-language note:

> The trap word generated by the _HOpen macro is the same as the trap
> word that would be generated by invoking the _Open macro with bit 9
> set. The setting of this bit tells the File Manager to expect a
> larger parameter block containing the additional fields (such as a
> directory ID) needed to handle a hierarchical directory volume.

Inside Macintosh gives each H-variant a distinct **name**, a distinct
**Pascal routine**, and a distinct **parameter block** (HParamBlockRec
vs ParamBlockRec).  They are real separate APIs in the programmer's
mental model, not internal flag variants.

### What about Memory Manager bit 9?

Bit 9 is reused by the Memory Manager for CLEAR (`_NewPtr ,CLEAR`).
Inside Macintosh does **not** document these as separate named
routines — there is no `NewPtrClear` Pascal function.  The docs show
them as macro flag variants:

```
_NewPtr ,CLEAR      (clears allocated block)
_NewPtr ,SYS,CLEAR  (applies to system heap and clears)
```

Since there's no distinct name in Inside Mac, these do **not** get
separate `traps.def` entries.  A future enhancement could display
the CLEAR/SYS bits as synthetic input parameters (e.g. `clear=YES`),
but that is out of scope here.

---

## Problem

`TrapDefs::maskTrapWord()` currently strips bits 9–10 from OS traps:

```cpp
return 0xA000 | (tw & 0x01FF);   // keeps bits 0-8 only
```

This maps $A014 and $A214 to the same key, so the tracer can only
carry one set of show-in/show-out fields.  HFS-specific fields like
`ioDirID` are invisible, and HOpen is displayed as just "Open".

---

## Solution: Preserve Bit 9 for OS Traps

Change `maskTrapWord()` to keep bits 0–9 for OS traps:

```cpp
return 0xA000 | (tw & 0x03FF);   // keeps bits 0-9
```

Flat and H-variant traps become distinct keys ($A014 ≠ $A214) while
bit 10 (ASYNC for File Manager, SYS for Memory Manager) is still
stripped — it doesn't change trap semantics or names.

Consequences:

- **HFS H-variant traps** get their own `traps.def` entries with
  HFS-appropriate struct types, names, and show fields.
- **`+Open` does not match `+HOpen`** — they are separate traps.
  This matches the Inside Mac mental model where they are distinct
  named APIs.
- **Memory Manager CLEAR variants** ($A31E for `_NewPtr ,CLEAR`)
  also become distinct keys, but we do NOT add entries for them.
  An unrecognized trap word simply traces as hex with no decoded
  fields — the same as today.  These will be addressed later with
  a flag-as-parameter mechanism.
- **`allowed_` bitset**: already `std::bitset<65536>`, no growth
  needed.

## 1. Code Changes

### 1.1 `maskTrapWord()` — `src/cpu/trap_defs.cpp`

```cpp
uint16_t TrapDefs::maskTrapWord(uint16_t tw)
{
    if (tw & 0x0800)
        return 0xA800 | (tw & 0x03FF);  // Toolbox: bits 0-9 + bit 11 (unchanged)
    else
        return 0xA000 | (tw & 0x03FF);  // OS: bits 0-9 (was 0x01FF)
}
```

One-line change.  Every call site (TrapDefs::load, TrapTracer::enter,
addTrap, removeTrap, addSubtrap, removeSubtrap) already passes through
this function — no other code changes needed for the mask itself.

### 1.2 Comment update — `assets/traps.def`

Update the existing comment about NewPtrClear:

```
# NewPtrClear ($A31E) = NewPtr ($A11E) + CLEAR bit (bit 9).
# OS trap masking now preserves bit 9, so $A31E and $A11E are
# distinct keys.  No separate entry is defined for NewPtrClear
# since Inside Mac does not give it a distinct name.
# Bit 10 (SYS heap) is still masked off.
```

## 2. New Struct Type

HOpen, HOpenRF, and HRename need both IO-style fields (`ioPermssn`,
`ioMisc`) and the HFS `ioDirID` at offset 48.  Neither `IOParam` nor
`HFileInfo` has both.

Add to `assets/types.def`:

```
struct HIOParam {
    0  ParamBlockHeader header
    24 sword    ioRefNum
    26 sbyte    ioVersNum
    27 sbyte    ioPermssn
    28 Ptr      ioMisc
    32 Ptr      ioBuffer
    36 long     ioReqCount
    40 long     ioActCount
    44 sword    ioPosMode
    46 slong    ioPosOffset
    48 long     ioDirID
}
```

`ioDirID` at offset 48 overlaps the tail of `ioPosOffset`.  This
matches HParamBlockRec from Inside Mac — the File Manager reuses
that space for H-variant calls.  The tracer only reads fields named
in show-in/show-out, so the overlap is harmless.

## 3. New `traps.def` Entries

### 3.1 HFS File Manager Variants

Each entry matches the name given by Inside Macintosh.  The struct
type is chosen to expose the HFS-specific fields at their documented
offsets in HParamBlockRec.

```
A200 HOpen os
  in  pb:^HIOParam.A0
  out err:OSErr.D0
  show-in  pb ioNamePtr ioVRefNum ioDirID ioPermssn
  show-out pb ioResult ioRefNum

A208 HCreate os
  in  pb:^HFileInfo.A0
  out err:OSErr.D0
  show-in  pb ioNamePtr ioVRefNum ioDirID
  show-out pb ioResult

A209 HDelete os
  in  pb:^HFileInfo.A0
  out err:OSErr.D0
  show-in  pb ioNamePtr ioVRefNum ioDirID
  show-out pb ioResult

A20A HOpenRF os
  in  pb:^HIOParam.A0
  out err:OSErr.D0
  show-in  pb ioNamePtr ioVRefNum ioDirID ioPermssn
  show-out pb ioResult ioRefNum

A20B HRename os
  in  pb:^HIOParam.A0
  out err:OSErr.D0
  show-in  pb ioNamePtr ioVRefNum ioDirID ioMisc
  show-out pb ioResult

A20C HGetFileInfo os
  in  pb:^HFileInfo.A0
  out err:OSErr.D0
  show-in  pb ioNamePtr ioVRefNum ioFDirIndex ioDirID
  show-out pb ioResult ioNamePtr ioFlAttrib ioFlFndrInfo ioFlNum ioFlLgLen ioFlRLgLen ioFlCrDat ioFlMdDat ioDirID ioFlParID

A20D HSetFileInfo os
  in  pb:^HFileInfo.A0
  out err:OSErr.D0
  show-in  pb ioNamePtr ioVRefNum ioDirID ioFlFndrInfo
  show-out pb ioResult

A214 HGetVol os
  in  pb:^WDParam.A0
  out err:OSErr.D0
  show-out pb ioResult ioNamePtr ioVRefNum ioWDProcID ioWDVRefNum ioWDDirID

A215 HSetVol os
  in  pb:^WDParam.A0
  out err:OSErr.D0
  show-in  pb ioNamePtr ioVRefNum ioWDDirID
  show-out pb ioResult
```

### 3.2 Memory Manager — No New Entries

Inside Macintosh documents `_NewPtr ,CLEAR` as a flag modifier on
the same trap macro, not as a separate named routine.  No entries
are added.  Future work may display CLEAR/SYS as synthetic input
parameters on the existing NewPtr/NewHandle entries.

## 4. What Does Not Change

- **`allowed_` bitset**: already `std::bitset<65536>`, no growth.
- **`addAllTraps()`**: calls `allowed_.set()` on all bits, works
  as-is.
- **Auto-pop detection**: tests `(tw & 0x0800) && (tw & 0x0400)` on
  the raw trapWord, not the masked value.  Unaffected.
- **Subtrap dispatch**: HFSDispatch ($A260) is an OS trap at index
  $060, well below the bit-9 boundary.  Unaffected.
- **Toolbox masking**: the Toolbox branch already keeps bits 0–9 +
  bit 11.  Unchanged.
- **The `+Name` / `-Name` filter syntax**: works on masked trapWords
  via the sorted name index.  H-variants are separate entries with
  separate names, so `+HOpen` enables HOpen and `+Open` enables flat
  Open.  `+All` still enables everything.

## 5. Trace Output Examples

### Before (flat Open used for both)

```
→ Open  pb=$0123F800
    pb^.ioNamePtr:  "\pSources:main.c"
    pb^.ioVRefNum:  -32001
    pb^.ioPermssn:  1
```

`ioDirID` is invisible — the tracer doesn't know it's an HFS call.

### After (H-variant has its own definition)

```
→ HOpen  pb=$0123F800
    pb^.ioNamePtr:  "\pmain.c"
    pb^.ioVRefNum:  -32001
    pb^.ioDirID:    54
    pb^.ioPermssn:  1
```

The trap name itself tells the reader it's HFS-aware, and `ioDirID`
appears in the field dump.

## 6. Testing

1. **Unit**: extend `test/debugger_smoke.sh` — verify `+HOpen` and
   `+Open` are independent filters that each resolve to their
   respective trap word.
2. **Manual**: boot with `--trace-traps +HGetVol +GetVol`, open a
   file from Finder.  Confirm HGetVol shows `ioWDDirID` while
   GetVol shows only `ioVRefNum`.
3. **Regression**: run with `--trace-traps +All` and confirm that
   `_NewPtr ,CLEAR` calls ($A31E) still trace without error (they
   just won't have decoded fields until the flag-as-parameter
   mechanism is implemented).
