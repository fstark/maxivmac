# INIT Versioning

The guest INIT reports its identity, version, and environment to the
host at boot time via a single register command.  The host overlay
displays the INIT status and detects stale builds.

## How it works

At boot, the INIT sends `kCmdInitIdent` ($0001) with seven parameters:

| Param | Content                                         |
|-------|-------------------------------------------------|
| p0    | `kApiVersion` (monotonic protocol version)       |
| p1    | Version string (Pascal, e.g. `"\pdev-a3b7c9f"`) |
| p2    | `vRefNum` of the INIT file                       |
| p3    | `dirID` of the INIT's parent directory            |
| p4    | Filename (Pascal, from `PBGetFCBInfo`)           |
| p5    | `machineType` (from `SysEnvirons`)               |
| p6    | `systemVersion` (BCD, from `SysEnvirons`)        |

The host checks `kApiVersion` against its supported range.  If
compatible, it returns 0 and the INIT proceeds to install traps and
the jGNEFilter unconditionally.  If incompatible, the INIT bails.

The old per-feature version checks (`ExtFSVersion` / `ClipVersion`)
are removed.  The host still handles those commands for backward
compatibility with older INITs.

## Overlay display

The overlay shows three states:

| State      | Display                                  | Color  |
|------------|------------------------------------------|--------|
| Not loaded | `INIT: not loaded`                       | gray   |
| Current    | `Mac II · System 6.0.7 · INIT dev-abc`  | white  |
| Stale      | `Mac II · System 6.0.7 · INIT dev-abc`  | yellow |

"Stale" means the INIT version string doesn't match the host's
`MAXIVMAC_VERSION`.

## Building the INIT

Run `./build-init.sh` from the repo root.  This:

1. Stamps `macsrc/init/version.h` with the current git version.
2. Launches maxivmac headless with `macsrc/build.mac` and the
   `build.dbg` script to compile the INIT inside THINK C.

The built INIT lands on the shared drive at `macsrc/init/`.

For manual builds, copy `version.h.default` to `version.h` — the
INIT reports `"dev"`.

| File                | Tracked | Purpose                          |
|---------------------|---------|----------------------------------|
| `version.h.default` | yes     | Default version for manual builds |
| `version.h`         | no      | Generated by `build-init.sh`      |

## Session reset

When `kCmdInitIdent` is received the host clears stale state in all
subsystems (KV store, clipboard cache, sequence numbers).  This
treats `InitIdent` as a session-reset signal — no stale clipboard
data leaks across guest reboots.

## Shift-skip

Standard Mac INIT convention: holding Shift at startup skips the
INIT.  The INIT checks the keyboard state early and bails with a
`dbg_log` message.  The overlay shows "INIT: not loaded".

## Future work

- **Auto-update:** The INIT reports its file spec (vRefNum, dirID,
  filename).  A future host feature could detect a stale INIT and
  write an updated copy to the guest disk automatically.
- **API version negotiation:** Currently the host accepts exactly
  `kApiVersion = 1`.  If register commands change semantics, bump
  `kApiVersion` and widen the host's accepted range.

### Resource ID 314 → 128

The INIT resource currently uses `GetResource('INIT', 314)` — a
legacy ID from the old SharedDrive-only INIT.  Move to ID 128
(standard for a single INIT resource) as part of this work.

### Clipboard errors are silent

`SyncClipboard()` returns void and ignores all errors from
`ExportMacToHost` / `ImportHostToMac`.  If clipboard transfer fails,
nothing is logged.  Add `dbg_log` calls on error paths so failures
are visible in the debug console.

## Future work (parked)

### Protocol cleanup — unify under `$0xx`

Several mechanisms currently scattered across `$1xx` and `$2xx` are
really system-level concerns:

| Current location | What | Should be |
|------------------|------|-----------|
| `$0108` (clip) `kClipDbgLog` | Debug logging | `$0xx` — single system log |
| `$0220` (drive) `kExtFSGuestCmd` | Launch, ExitToShell, Shutdown | `$0xx` — system commands |
| jGNEFilter polls `PollMounts` + `PollGuestCmd` separately | Host→guest channel | `$0xx` — single poll, dispatch by command type |

Target architecture:
- **Single system log** in `$0xx`, used by system, clip, and drive.
- **Single host→guest command channel** in `$0xx`.  The guest polls
  one command, gets a command code + payload, and dispatches to the
  appropriate subsystem (system, clip, drive).
- **System commands** (launch, ExitToShell, shutdown) move to `$0xx`.
- `$1xx` and `$2xx` become pure feature protocols with no system
  plumbing mixed in.

### Auto-update

The INIT's file spec (vRefNum, dirID, name) is already captured at
boot.  A future "copy file to guest" command could overwrite the INIT
file on the user's boot disk and trigger a reboot.  The INIT file is
not open after boot (the System closes it after loading resources), so
overwriting it is safe.

### `vers` resource self-stamping

The INIT could write a `vers` resource into its own file at first boot
using the captured `HomeResFile` refnum, so Finder's Get Info shows
the correct version.  Since `kInitVersion` is already compiled in, no
additional data transfer is needed.

### Generic file transfer

A "write file to guest path" command would enable drag-and-drop from
host to guest desktop.  This is a separate feature that shares the
chunked register-block transfer mechanism but differs in targeting
(arbitrary path vs. INIT's own file).
