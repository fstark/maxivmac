# LATER — Non-Critical Differences vs Reference

These differences were found by systematically comparing the current
maxivmac code against the original minivmac reference (configured as a
Mac Plus with `vMac.ROM`). None are believed to cause crashes, but they
should be addressed for correctness, maintainability, or parity when
targeting other Mac models.

The reference configuration was generated by `build-reference.sh` which
runs `reference/setup_t` → `reference/setup.sh` → overwrites
`reference/cfg/CNFUDPIC.h`. The values below are from the **actual build
output**, not the stale checked-in cfg file.

---

## 1. SubTickNotify dangling-else bug (Mac II / ASC models)

**Current:** `src/core/main.cpp`, lines ~101–106

```cpp
if (g_machine->config().emClassicSnd)
    if (auto* d = g_machine->findDevice<SoundDevice>()) d->subTick(SubTick);
else if (g_machine->config().emASC)
    if (auto* d = g_machine->findDevice<ASCDevice>()) d->subTick(SubTick);
else
    UnusedParam(SubTick);
```

The `else if (emASC)` binds to the **inner** `if (auto* d = ...)`, not the
outer `if (emClassicSnd)`. When `emClassicSnd=false` (Mac II, IIx), the
outer `if` is false, and the entire chain is skipped — ASC subtick
processing never fires.

**Reference:** `reference/src/PROGMAIN.c`, lines ~115–122

```c
#if EmClassicSnd
    MacSound_SubTick(SubTick);
#elif EmASC
    ASC_SubTick(SubTick);
#else
    UnusedParam(SubTick);
#endif
```

Clean preprocessor selection, no ambiguity.

**Impact on Plus:** None (`emClassicSnd=true`).  
**Impact on Mac II/IIx:** ASC sound subticks never fire. No sound output.

**Fix:** Add braces around the outer `if` body, or restructure as
`if/else if/else` with braces.

## 2. `autoSlowTime` — 60 vs 34

**Current:** `src/core/machine_config.h`, line ~86 — default `autoSlowTime = 60`, never overridden for any model.  
**Reference (Plus build):** `kAutoSlowTime = 34` (generated by `setup.sh`, line 266).

The auto-slow mechanism kicks in after N ticks of idle emulation to
reduce host CPU usage. The current version waits 60 ticks (~1 second)
vs the reference's 34 ticks (~0.5 seconds).

**Impact:** No correctness issue. The emulated Mac runs identically; only
host-side power consumption differs during idle.

**Fix:** Set `autoSlowTime = 34` for the Plus model in
`machine_config.cpp` (around line 279), or make it configurable per model.

## 3. `extnBlockBase` — 0x00F0C000 vs 0x00F40000

**Current:** `src/core/machine_config.cpp`, line ~307 — `extnBlockBase = 0x00F0C000` for all compact Macs.  
**Reference (Plus build):** `kExtn_Block_Base = 0x00F40000` (generated by `setup.sh`, line 404).

These addresses are both in unused address space (above ROM, below 24-bit
wrap). The value only matters for internal consistency: the ATT mapping
and the ROM patch must agree. Both versions are self-consistent.

**Impact:** None — the specific address is arbitrary as long as nothing
else maps there. However, for trace-level determinism, the different
addresses would show up in any Extn_Access I/O log comparison.

**Fix:** Change to `0x00F40000` for Plus/SE/Classic models to match the
reference, or keep current value and document the intentional difference.

## 4. RAM bank split — single 4MB vs two 2MB banks

**Current:** `src/core/machine_config.cpp`, line ~302 — `ramASize = 0x00400000, ramBSize = 0` (single 4MB bank).  
**Reference (Plus build):** `kRAMa_Size = 0x00200000, kRAMb_Size = 0x00200000` (two 2MB banks, generated by `setup.sh`, lines 268-269).

Both produce **identical** ATT mapping results: 4MB total with
`usemask = 0x003FFFFF`. The single-bank path `(ramBSize == 0)` in
`SetUp_RAM24_compact()` produces the same ATT entry as the two-equal-banks
path in the reference.

**Impact:** None for Mac Plus. Could matter if we later emulate a Mac with
asymmetric RAM banks (e.g., Mac 128K with expansion), but currently both
paths collapse to the same result.

**Fix:** Change Plus config to `ramASize = 0x00200000, ramBSize = 0x00200000`
to precisely match the reference, or keep current value since behavior is
identical.

## 5. PRAM initial values differ

**Current:** `cfg/CNFUDPIC.h`  
**Reference (Plus build):** values from `setup.sh`

| Parameter | Current | Reference | Notes |
|-----------|---------|-----------|-------|
| `CaretBlinkTime` | `0x08` | `0x03` | Cursor blink rate |
| `DoubleClickTime` | `0x08` | `0x05` | Double-click speed threshold |

These are initial PRAM parameter values written into emulated PRAM at
startup. They affect the user experience (blink speed, double-click
sensitivity) but not emulation correctness.

**Impact:** Cosmetic — the emulated Mac feels slightly different to use
(slower blink, more forgiving double-click window).

**Fix:** Change `CaretBlinkTime` to `0x03` and `DoubleClickTime` to `0x05`
in `cfg/CNFUDPIC.h` to match the reference.

## 6. Dead extern declarations in VIA code

**Current:** `src/devices/via.cpp`, lines ~38–69

```cpp
extern void VIA1_iA0_ChangeNtfy();
extern void VIA1_iA1_ChangeNtfy();
// ... through iA7, iB0, iB1
```

These `extern` declarations reference functions from the old C codebase's
wire-change notification system. The current version uses the `WireBus`
callback mechanism instead. These externs are never called and likely
don't even link (the functions don't exist in the C++ build).

**Impact:** None — dead code.

**Fix:** Remove the dead extern declarations.

## 9. AutoSlow disabled by determinism patch

**Current:** `src/core/main.cpp` `MoreSubTicksToDo()` and
`reference/src/PROGMAIN.c` `MoreSubTicksToDo()` — the `ExtraTimeNotOver()`
wall-clock gate has been removed so that all extra sub-ticks always
complete, making the emulated cycle count per tick deterministic across
runs.

**Side-effect:** The `#if EnableAutoSlow` idle-detection code inside
`MoreSubTicksToDo` was also removed. AutoSlow normally detects host-side
idleness (no keyboard/mouse/screen activity for ~0.5 s) and drops back
to 1x speed to save CPU. With the patch, extra sub-ticks always run
regardless of idle state.

**Fix:** Re-introduce the AutoSlow guard *without* `ExtraTimeNotOver()`:
```c
if (ExtraSubTicksToDo > 0) {
#if EnableAutoSlow
    if (idle-check) { ExtraSubTicksToDo = 0; }
    else
#endif
    { v = true; }
}
```

This preserves determinism while restoring the power-saving behaviour.

fred's note: I do think the way the emulator manage speed is wrong. Need to check the exact behavior (in particular the clock) to see how to make something less convoluted.

## 10. LeaPeaEACalcCyc return type bug (inherited from reference)

**File:** `src/cpu/m68k_tables.cpp`, line ~491

The reference declares `LeaPeaEACalcCyc` as returning `blnr` (`uint8_t`),
but uses `ui4r` (`uint16_t`) values internally. Values above 255 silently
wrap — e.g. mode 7 reg 2 (PC+d16) computes `4*64+80 = 336` which wraps to
`80`. The C++ port initially used `bool`, clamping any non-zero to `1`,
which was even worse.

**Current fix:** Changed return type to `uint8_t` to match the reference's
truncation behavior and achieve cycle-exact parity.

**Proper fix:** Change both the reference and debug return types to
`uint16_t` and accept the (correct) larger cycle costs. This would improve
timing accuracy for LEA/PEA instructions but would intentionally break
parity with the original minivmac 3.7.
